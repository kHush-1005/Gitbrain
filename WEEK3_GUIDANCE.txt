WHAT WEEK 3 BUILDS

Week 1 fetched code from GitHub.
Week 2 chunked that code and saved it as JSONL files.
Week 3 converts those chunks into vector embeddings and stores them in
ChromaDB so you can search the codebase with natural language.

After Week 3:

This is semantic search — finding code by MEANING, not by keywords.


────────────────────────────────────────────────────────────────────────────────
KEY CONCEPTS (EXPLAINED SIMPLY)
────────────────────────────────────────────────────────────────────────────────

WHAT IS AN EMBEDDING?
  An embedding is a list of 384 numbers that represents the "meaning" of
  a piece of text. The sentence-transformers model converts any text into
  this list of numbers. Similar texts produce similar number lists.

  Example:
    "user login authentication"  → [0.12, -0.34, 0.78, ...]  (384 numbers)
    "how does user auth work?"   → [0.11, -0.31, 0.79, ...]  (very similar)
    "database table schema"      → [-0.55, 0.23, -0.12, ...]  (very different)

WHAT IS CHROMADB?
  ChromaDB is a local vector database. It stores:
    - The chunk text (what the LLM will read in Week 4)
    - The embedding vector (what gets searched)
    - The metadata (file path, line numbers, chunk type)

  When you search, ChromaDB compares your query vector against all stored
  vectors and returns the closest matches.

  All data is saved on disk in: data/chroma_db/
  No cloud account needed. No cost. Free.

WHAT IS COSINE SIMILARITY?
  ChromaDB measures similarity using cosine distance.
  We convert it to a score between 0.0 and 1.0:
    1.0 = identical  (same text)
    0.7+ = very similar (same topic)
    0.4–0.6 = related
    < 0.3 = probably unrelated

────────────────────────────────────────────────────────────────────────────────
FIRST-TIME MODEL DOWNLOAD
────────────────────────────────────────────────────────────────────────────────

The first time you use the embedder, it downloads the all-MiniLM-L6-v2 model
(~22 MB) from Hugging Face. This happens automatically.

  Where it's saved:
    Windows: C:\Users\<you>\.cache\huggingface\hub\
    Linux:   ~/.cache/huggingface/hub/

  After the first download, every subsequent run uses the cached version
  — no internet connection needed.

If HuggingFace is blocked on your network:
  Download the model manually from: https://huggingface.co/sentence-transformers/all-MiniLM-L6-v2
  Or use a different model with the --model flag.

────────────────────────────────────────────────────────────────────────────────
WHAT EACH NEW FILE DOES
────────────────────────────────────────────────────────────────────────────────

core/embedder.py
  Wraps the sentence-transformers library.
  Key functions:
    get_embedding_model()  — lazy singleton; loads model once, reuses
    embed_text(text)       — single text → list of 384 floats
    embed_batch(texts)     — list of texts → list of 384-float lists
    validate_embedding(v)  — check that a vector has correct dimension/type

core/vector_store.py
  Wraps ChromaDB for storing and searching embeddings.
  Key functions:
    get_client(dir)                       — lazy ChromaDB client singleton
    sanitize_collection_name(name)        — make name ChromaDB-safe
    get_or_create_collection(name, dir)   — open or create a collection
    upsert_chunks(chunks, name)           — store embedded chunks
    similarity_search(vector, name, k)    — find top-k similar chunks
    collection_count(name)                — count stored chunks
    delete_collection(name)               — delete for re-indexing

scripts/build_vector_index.py
  Main Week 3 script. Runs: load JSONL → embed → store in ChromaDB.
  Accepts --batch-size, --fresh, --repo-name flags.

scripts/semantic_search.py
  Query the vector index from terminal.
  Accepts --top-k, --language, --min-score, --show-code flags.

tests/test_embedder.py
  25 pytest tests covering: dimension check, float types, determinism,
  semantic similarity sanity, empty input error handling.
  (Requires HuggingFace model download on first run)

tests/test_vector_store.py
  38 pytest tests covering: name sanitization, collection CRUD,
  upsert/count/search/delete, metadata preservation, edge cases.
  (Uses fake [0.1]*384 embeddings — no model needed — runs offline)

COMMON WINDOWS ERRORS AND FIXES
────────────────────────────────────────────────────────────────────────────────

ERROR: 'venv\Scripts\activate' is not recognized
  FIX: Make sure you are in the gitbrain\ folder.
       Run: dir  to confirm venv\ folder exists.
       If not: python -m venv venv  then try again.

ERROR: sentence-transformers install is very slow
  FIX: It is downloading PyTorch (~300-800 MB). This is normal.
       Leave it running. It happens once.

ERROR: ERROR: Could not find a version of torch
  FIX: pip install torch --index-url https://download.pytorch.org/whl/cpu
       Then: pip install sentence-transformers

ERROR: OSError: [Errno 28] No space left on device
  FIX: Free up disk space. PyTorch + model needs ~2 GB free.
       Delete old Downloads or temporary files.

ERROR: RuntimeError: Failed to load embedding model
       "couldn't connect to 'https://huggingface.co'"
  FIX: Check your internet connection.
       If HuggingFace is blocked on your network:
         - Use a VPN
         - Or download the model on another machine and copy the cache folder

ERROR: FileNotFoundError: JSONL file not found
  FIX: Run Week 2 first:
       python scripts\export_chunks.py https://github.com/psf/requests
       Then run build_vector_index.py again.

ERROR: Collection 'X' is empty after indexing
  FIX: Check that the JSONL file has content:
       python scripts\validate_chunks.py data\chunks\X_chunks.jsonl
       If empty: re-run export_chunks.py

ERROR: Wrong file path separators (/ vs \)
  FIX: The scripts use pathlib internally and accept both / and \.
       Both of these work on Windows:
         python scripts\build_vector_index.py data\chunks\psf__requests_chunks.jsonl
         python scripts\build_vector_index.py data/chunks/psf__requests_chunks.jsonl

ERROR: Collection name mismatch in semantic_search.py
  FIX: The repo name in semantic_search.py must match the collection name
       that was created by build_vector_index.py.
       Run: python -c "from core.vector_store import list_collections; print(list_collections())"
       To see all available collection names.

ERROR: chromadb.errors.InvalidCollectionException
  FIX: The collection name has illegal characters.
       The sanitize_collection_name() function handles this automatically.
       If you passed a custom --repo-name, verify it has only a-z, 0-9, _.

WEEK 4 PREVIEW
────────────────────────────────────────────────────────────────────────────────

After Week 3, GitBrain can find relevant code chunks for any natural language
query. Week 4 connects this to an LLM to turn those chunks into answers.

Week 4 builds:
  core/llm_client.py     — wraps Groq API (Llama 3.1, free tier)
  core/rag_engine.py     — orchestrates: embed query → retrieve → build prompt → LLM
  api/main.py            — FastAPI with /ingest and /query endpoints
  api/routes/ingest.py   — POST /ingest triggers ingestion + indexing
  api/routes/query.py    — POST /query returns LLM answer + citations

New packages for Week 4:
  pip install fastapi uvicorn groq

You will need a free Groq API key from: https://console.groq.com
Add it to .env: GROQ_API_KEY=gsk_yourkey

Week 4 query flow:
  User asks: "how does authentication work?"
  1. embed the question (embedder.py)
  2. retrieve top-5 chunks (vector_store.py)  ← BUILT IN WEEK 3
  3. build a prompt with the chunks + question (rag_engine.py)
  4. send prompt to Llama 3.1 via Groq (llm_client.py)
  5. return answer + source citations (FastAPI /query endpoint)
